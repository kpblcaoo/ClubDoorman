# ClubDoorman.Baseline

Изолированный консольный harness для генерации Golden Master снапшотов (фазы 0‑3).

## Эволюция фаз
| Фаза | Суть | Артефакты |
|------|------|-----------|
| 0 | Freeze первичного набора сценариев | `golden/freeze-manifest.json` |
| 1 | Семантическое обогащение: добавление `ruleCode` (детерминированная классификация причин) | Обновлённые `*.output.json` |
| 2 | Манифест сценариев (детерминированная индексация) | `golden/manifest.json` (Schema=1) |
| 3 | Параллельный экспорт упрощённых V2 снапшотов | `golden/baseline_v2/*.v2.json` (Schema=2) |

Дальнейшие фазы (план): нормализация ID, инварианты, агрегированные метрики, diff‑gate в CI, удаление v1 после стабилизации.

## Цели
* Нет baseline / golden логики в production `Program.cs`.
* Детерминированные correlationId и последовательные числовые Id (1..N) в манифесте.
* Полное покрытие манифеста: ни одного `Unknown` `ruleCode`.
* Параллельный экспорт V2 (минимальное представление: Schema/CorrelationId/Id/ShortName/Action/RuleCode) без дублирования вычислений.

## Запуск
```
dotnet run --project ClubDoorman.Baseline
```
Параметр `DOORMAN_BASELINE_MEDIA_OFF=1` создаст дополнительный вариант `baseline_mediaoff` (пока манифест генерируется только для основного `baseline`).

## Результат
В каталоге `golden/`:
* `baseline/*.input.json` – входные апдейты
* `baseline/*.output.json` – детальный v1 вывод
* `manifest.json` – список сценариев (Schema=1)
* `baseline_v2/*.v2.json` – упрощённые v2 снапшоты (Schema=2)

## Добавление сценария
1. Добавьте synthetic update в `GoldenBaselineSeeder` (сохраняйте монотонность updateId).
2. Прогоните baseline harness – новые файлы появятся автоматически, манифест регенерируется.
3. Проверьте тесты `GoldenManifestTests` и `GoldenV2ExportTests` – они обеспечат непрерывность ID и синхронизацию содержимого.

## Инварианты, которые тестируются
* IDs в манифесте: плотный диапазон 1..N.
* Каждому output соответствует запись в манифесте (нет сирот).
* `ruleCode != Unknown` и при `ruleCode=Pass` ожидаемое `ExpectedAction=Allow`.
* Снимки V2 синхронизированы по (Id,CorrelationId,ShortName,Action,RuleCode) с манифестом.

## Почему V2?
V1 хранит полный «сырые» поля для будущей диагностики. V2 — минимальное стабильное представление для лёгкого сравнения и будущих агрегатов. Это позволяет менять внутреннюю структуру v1 без ломки контрактов сравнения.

## Правила модификации
* Не переставлять существующие Id – добавлять только новые в конец последовательности (иначе нарушатся исторические диффы).
* Новые `RuleCode` добавлять в enum в конце (стабильность численных значений на будущее).
* Любое появление `Unknown` считать регрессионным сигналом – обновить маппер.

## Очередные шаги (Roadmap)
1. Phase 4: нормализация переиспользуемых полей (абстракция внутренних динамических значений).
2. Phase 5: вынесение reason provider / генерация инвариантов.
3. Phase 6: агрегаты (coverage по RuleCode, распределение действий) как отдельный JSON.
4. Phase 7: CI diff gate (падение при нежданных изменениях без обновления манифеста).
5. Phase 8: удаление v1 после периода стабильности.

## Замечание
Все baseline‑специфичные классы (сидер, offline Telegram, экспорт) живут только здесь и не «подтекают» в основной сервис.
