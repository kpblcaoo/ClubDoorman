# Golden Master / Baseline System

(Документация для разработчиков: внутренняя архитектура, флаги, как снимать и поддерживать baseline.)

## Цели
Golden Master (GM) подсистема решает три задачи:
1. Зафиксировать фактическое поведение критических обработчиков (вход / выход) для будущих рефакторингов.
2. Снизить риск регрессий, которые не покрывают обычные unit/integration тесты.
3. Дать разработчику быстрый визуальный дифф по «что реально приходит / уходит» после изменений.

## Что пишется
Для каждого сэмпла (по выборке `GoldenSampleRate`) пишутся два JSON файла:
- `<correlationId>.input.json` — нормализованное входное Update (маскированы userId / username, усечён текст, null'ы удалены, свойства отсортированы).
- `<correlationId>.output.json` — результат высокоуровневой обработки (то что передали в `TryRecordOutput`).

## Маскирование и приватность
- UserId → `Uxxxx` (остаток mod 10000, с ведущими нулями)
- Username → `u_` + `GetHashCode()` (hex)
- Текст сообщения обрезается до 160 символов.
- Null-свойства удаляются.
- Свойства сортируются лексикографически → детерминированный diff.

## Флаги (секция `LoggingFlags` / ENV префикс `LoggingFlags__`)
| Параметр | Тип | Назначение | По умолчанию |
|----------|-----|------------|--------------|
| `TraceEnabled` | bool | Доп. информационные логи (категория trace) | false |
| `GoldenMasterEnabled` | bool | Включить запись GM | false |
| `GoldenSampleRate` | double 0..1 | Доля Update'ов для записи | 0.1 |
| `GoldenBasePath` | string | Корневая папка для файлов | `golden` |
| `GoldenDeterministicIds` | bool | Детерминированный correlationId (baseline) | false |
| `GoldenFixedDateFolder` | string? | Фиксированная подпапка вместо yyyy-MM-dd | null |

### Формат correlationId
- Обычный режим: SHA256(`update.Id:handler:timestamp:guid`) → первые 16 hex.
- Детерминированный режим: SHA256(`update.Id:handler:update.Type:chatId:userId`) → первые 16 hex.

## Структура каталогов
```
 golden/
   2025-08-18/          # обычные ежедневные (рабочий режим)
     ab12cd34ef56.input.json
     ab12cd34ef56.output.json
   baseline/            # фиксированная подпапка с эталоном
     9f3a0b1c2d3e.input.json
     9f3a0b1c2d3e.output.json
```

## Снятие baseline
Используем встроенный скрипт:
```
./scripts/generate_golden_baseline.sh --clean
```
Скрипт выставляет:
```
LoggingFlags__GoldenMasterEnabled=true
LoggingFlags__GoldenSampleRate=1.0
LoggingFlags__GoldenDeterministicIds=true
LoggingFlags__GoldenFixedDateFolder=baseline
```
Baseline перегенерируется в `golden/baseline`. Просмотри файлы, убедиcь в отсутствии PII и закоммить.

## Когда обновлять baseline
Обновляй baseline ТОЛЬКО когда произошло сознательное изменение интерфейса обработчика (новые поля в output, другое нормализованное поведение). В PR:
1. Сначала изменения кода.
2. Отдельный коммит: переснятый baseline.
3. В описании — краткое «что изменилось и почему корректно».

## Проверка локально
1. Запусти тесты: `dotnet test` (GM тест проверяет детерминизм / канонизацию).
2. При включённом GM (не baseline режим) убедись что появляются ежедневные папки.
3. Сравни diff baseline после твоих прав — не должно быть лишних шумов.

## Типичные сценарии
| Ситуация | Действие |
|----------|----------|
| Рефактор без изменения семантики | baseline НЕ трогать; при появлении diff → расследовать почему изменился snapshot |
| Добавил новое поле в output `TryRecordOutput` | Переснять baseline, описать поле в PR |
| Добавил новый вид Update (например, CallbackQuery instrumentation) | Переснять baseline после внедрения |
| Утечка PII обнаружена | Исправить маскирование → переснять baseline |

## Расширение схемы snapshot
При добавлении новых данных старайся:
1. Маскировать всё, что может идентифицировать (ID → сокращённый, строки → хэш/усечение при необходимости).
2. Не логировать большие payload'ы (ограничение длины).
3. Сначала покрыть новым тестом, затем расширить snapshot.
4. В output предпочтительно писать компактный объект (action, reason, flags), избегая дублирования входных полей.

## Тесты
- `GoldenMasterRecorderTests` — проверка детерминизма (два прогона → одинаковый input), отсутствия сырых ID / username.
- При добавлении новых полей в Canonicalize или SimplifyUpdate — расширить тест: проверка маски и сортировки.

## Потенциальные ловушки
| Проблема | Симптом | Решение |
|----------|---------|---------|
| Слишком мало snapshot'ов | Почти пустая дата-папка | Увеличь `GoldenSampleRate` временно или force-run сценарии тестов / интеграций |
| Шум в diff (меняются correlationId) | Каждый commit меняет baseline | Убедись что `GoldenDeterministicIds=true` и `GoldenFixedDateFolder=baseline` при генерации baseline |
| Лишний PII | Видны реальные userId / username | Проверить маску в `SimplifyUpdate`, добавить тест на маскирование |
| snapshot папки копятся | Много дат в `golden/` | Добавить ручную или CI очистку (оставлять N последних) |

## Тонкости внедрения новых обработчиков
Чтобы добавить GM в новый handler:
```csharp
string? corr = _gm.TryRecordInput(update, nameof(NewHandler), chatId, userId);
// ... обработка ...
_gm.TryRecordOutput(corr, new { kind = "...", result = status });
```
Правила:
- Не бросать исключения из instrumentation (оборачиваем в try/catch).
- Output должен быть максимально компактным (для диффов).

## FAQ
**Q:** Почему не храним raw Update?  
**A:** Слишком много шума (динамические поля Telegram), риск PII, плохие диффы.

**Q:** Почему сортировка JSON вместо собственной схемы?  
**A:** Быстрая реализация, минимизирует случайные перестановки, сохраняет человекочитаемость.

**Q:** Чем baseline отличается от ежедневных папок?  
**A:** Baseline — один эталон; ежедневные — потоковая телеметрия / отладка.

**Q:** Что если baseline стал слишком маленьким / не покрывает сценарий?  
**A:** Добей сгенерированием дополнительных сцен, зафиксируй их и пересними baseline.
