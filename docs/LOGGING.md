# Логирование (Logging)

Этот документ описывает систему логирования Spampyre / ClubDoorman: уровни, конфигурацию через переменные среды, Golden Master события, области (scopes) и типичные сценарии использования.

## Цели логирования
- Диагностика инцидентов (ошибки, неожиданные ветки).
- Аудит действий модерации (капча, бан, одобрение, AI‑ограничения).
- Наблюдаемость ML/AI (обучение, классификация, fallback).
- Golden Master (GM) семантика — файловый baseline для регрессий *не дублируется* в бизнес‑логах сверх необходимого.

## Текущая архитектура
Serilog с несколькими sink:
- Консоль (уровень = глобальный минимум)
- `logs/clubdoorman-<date>.log` — общий поток
- `logs/errors-<date>.log` — только Error+
- `logs/system-<date>.log` — Information+ (системный слой Microsoft/System)
- `logs/userflow-<date>.log` — события с добавленным свойством `UserFlow` (отдельный фильтр)

## Уровни (LogLevel)
Используются стандартные уровни:
- TRACE — детальные переходы и промежуточные состояния (может быть выключен в бою)
- DEBUG — ключевые технические шаги (получен апдейт, вызов модерации, результат классификации)
- INFORMATION — бизнес‑события (прошёл капчу, забанен, одобрен, AI restricted)
- WARNING — восстановимые аномалии / деградация (внешний сервис недоступен, не найден объект для удаления, fallback)
- ERROR — операция не выполнена; потеря функциональности / критично для сценария

## Переменные среды для управления логами
| Переменная | Значение | По умолчанию | Описание |
|-----------|----------|--------------|----------|
| `DOORMAN_LOG_LEVEL` | Verbose|Debug|Information|Warning|Error|Fatal | Information | Базовый минимум логов приложения. |
| `DOORMAN_TRACE_ENABLE` | true/false | false | Если true — принудительно включает Verbose (Trace) независимо от `DOORMAN_LOG_LEVEL` (если тот выше). |

Пример включения полного трассинга локально:
```
DOORMAN_LOG_LEVEL=Debug
DOORMAN_TRACE_ENABLE=true
```
В продакшене обычно:
```
DOORMAN_LOG_LEVEL=Information
DOORMAN_TRACE_ENABLE=false
```

## Области (BeginScope)
Handlers добавляют в scope:
- `opId` — уникальный Guid операции
- `updateType` — тип апдейта Telegram
- `chatId`, `userId`, `messageId` — контекст при наличии
- `gmCorrelation` — идентификатор Golden Master корреляции (если включена запись семантики)

Все свойства становятся частью каждой строки лога внутри области — упрощает фильтрацию (напр. по конкретному `chatId`).

## Golden Master Moderation Events
Путь: `GoldenMasterModerationEventPublisher`.
- Паблишер получает `ModerationEvent` и корреляцию.
- Если корреляции нет — лог Trace `GM Event skipped…`.
- Если есть — одна строка `DEBUG` (summary) + детальный JSON payload на уровне `TRACE` (если включён).
- Затем payload пишется в Golden Master файлы через `IGoldenMasterRecorder`.

Так мы ограничиваем шум: без TRACE JSON не дублируется в общий лог.

## Типовые фильтры
Примеры (в зависимости от системы агрегации):
- Все ошибки конкретного чата: `chatId = -1001234567890 AND Level >= Error`
- Бизнес события пользователя: `userId = 12345 AND Level = Information`
- GM события по корреляции: `gmCorrelation = 91f9c2*`

## Рекомендации по использованию уровней
| Ситуация | Уровень |
|----------|---------|
| Ветка условия, быстрый early-exit | Trace |
| Вызов внешнего сервиса, пред и пост результат | Debug |
| Пользователь прошёл / не прошёл капчу | Information |
| Автобан / ручной бан / одобрение | Information |
| Fallback логика, пропуск из-за конфигурации | Warning |
| Исключение при критичном действии | Error |

## Частые ошибки
- Дублировать одно и то же событие на разных уровнях подряд. (Избегать.)
- Логировать полный текст длинного сообщения вне необходимости (при добавлении trimming — обрезать свыше разумного порога).
- Использовать Warning для штатных веток.

## Как снизить шум без релиза кода
Измените только переменные среды и перезапустите сервис:
```
DOORMAN_LOG_LEVEL=Warning
DOORMAN_TRACE_ENABLE=false
```
Это мгновенно сокращает объём файлов.

## Отладка локально
1. Поставьте `DOORMAN_TRACE_ENABLE=true` — получите полный Trace + JSON GM payload.
2. Воспроизведите сценарий (например, вход нового пользователя).
3. Фильтруйте по `gmCorrelation` → просматривайте поездку события.

## FAQ
**Q:** Почему не один файл логов?  
**A:** Разделение ошибок и userflow упрощает чтение и позволяет ротации по разным retention.

**Q:** Почему не structured JSON sink?  
**A:** Файловый лёгкий текст проще для локальной отладки. Можно добавить JSON sink позднее.

**Q:** Как увидеть только модерационные действия?  
**A:** Использовать фильтр по ключевым шаблонам ("CAPTCHA"/"банен"/"restricted") или реализовать дополнительный sink (план итерации 2).
